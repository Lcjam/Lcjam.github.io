백엔드 개발자로 길게 가려면 전공 지식이 비는 게 계속 걸렸다. 나는 심리학을 전공했고, KDT 부트캠프로 개발에 입문한 비전공자다. 부트캠프에서 당장 코드 짜는 법은 배웠지만, 그 아래에 깔린 컴퓨터과학을 제대로 짚고 넘어간 적은 없었다. 그 공백을 메우려고 한국방송통신대학교 컴퓨터과학과 3학년 1학기로 편입했다.

직장이랑 병행하는 게 쉽진 않다. 그래도 퇴근 후 공부는 개발자로 사는 한 계속 가져갈 루틴이라 크게 특별한 일이라고 생각하진 않는다. 정보처리기사나 ADsP 준비하면서 만들어 둔 공부 패턴이 있어서, 그걸 그대로 굴려 첫 학기는 무사히 끝냈다. (성적은 나오면 따로 공개하겠다.)

## 이산수학, 처음엔 좀 헷갈렸다

1학기 첫 과목이 이산수학이었다. 시니어 개발자들이 CS의 근본이라고 입을 모으던 과목이라 기대가 컸다.

그런데 막상 초반 단원 — 명제 논리, 증명, 집합, 행렬 — 을 보고 있으니 솔직히 좀 헷갈렸다. 내용 자체는 고등학교 수학의 연장처럼 느껴지는데, "실무에서 이걸 어떻게 활용한다는 거지?"가 강의만 들어서는 잘 안 이어졌다. 이론을 그냥 받아 적기만 하면 금방 지루해지는 구간이었다.

그러다 태도를 바꿨다. 교과서는 정리된 지식만 줄 뿐이고, 그걸 실무랑 연결하는 건 결국 내 몫이라는 걸 받아들였다. 그 뒤로는 개념 하나를 배우면 "이걸 서버 로직이나 데이터 처리에 갖다 붙이면 어떻게 될까?"를 일부러 한 번씩 떠올려봤다. 그러자 공식처럼만 보이던 게 조금씩 코드로 보이기 시작했다.

## 배운 걸 실무에 붙여봤다

사실 여기 나오는 코드들은 이산수학을 배우기 전에 팀프로젝트에서 먼저 짜둔 것들이다. 그땐 "이렇게 하면 되겠지" 하고 감으로 구현했는데, 수업을 듣고 다시 보니 각각 단방향 함수, 유한 상태 기계라는 이름이 붙는 것들이었다. 이론이 내가 감으로 한 걸 뒤늦게 언어로 정리해준 셈이다. 한 번 그렇게 추상화해 두니, 다음에 비슷한 설계를 할 땐 처음부터 그 틀로 더 빠르게 생각할 수 있겠다는 감이 왔다.

한눈에 정리하면 이렇다.

| 내가 짠 것 | 이산수학 개념 |
| :-- | :-- |
| 비밀번호 해싱 (BCrypt) | 단방향 함수 (one-way function) |
| 로그인 실패 잠금 | 유한 상태 기계 (FSM) |

이 두 개를 풀어보면,

### 비밀번호 해싱 = 단방향 함수

회원 비밀번호는 BCrypt로 해싱해서 저장한다.

```java
var encoder = new BCryptPasswordEncoder(12);
String hashed = encoder.encode(password);              // 저장할 때
boolean ok    = encoder.matches(inputPassword, hashed); // 검증할 때
```

이게 이산수학에서 말하는 **단방향 함수(one-way function)** 다. 평문 → 해시는 계산이 쉽지만, 해시 → 평문(역함수)은 사실상 계산이 불가능하다. 그래서 DB가 통째로 털려도 원문 비밀번호는 복구가 안 된다.

> **단방향 함수** — 한쪽(평문 → 해시) 계산은 쉽지만, 역방향(해시 → 평문)은 사실상 계산이 불가능한 함수.
{: .prompt-info }

검증할 때 `matches`를 쓰는 이유도 여기서 나온다. 복호화가 안 되니 "풀어서 비교"는 불가능하고, 대신 저장된 해시 안에 들어 있는 salt로 입력을 똑같이 해싱해서 결과끼리 비교한다. salt를 섞는 건 같은 비밀번호라도 사람마다 다른 해시가 나오게 하려는 거다 — 미리 계산해 둔 해시표(레인보우 테이블) 공격이 안 통하는 이유다.

생성자의 `12`도 그냥 강도 숫자가 아니라 2¹² = 4096회 반복(키 스트레칭)을 뜻한다. 숫자를 1 올릴 때마다 연산량이 두 배가 되는 지수 증가라, 공격자의 brute-force 비용을 일부러 키운 장치다.

### 로그인 실패 잠금 = 유한 상태 기계

로그인 5회 실패하면 계정을 잠그는 로직도, 보고 나니 **유한 상태 기계(FSM)** 그 자체였다.

- 상태 집합: `{ Active, Locked }`
- 전이: 실패하면 카운트 +1 → 카운트가 5에 닿으면 `Active → Locked`. 로그인 성공하면 카운트를 리셋하며 `Active` 유지.

```text
       실패 → count++ (5 미만이면 그대로)
      ┌───────────────────────────────┐
      ▼                               │
 ┌──────────┐    count >= 5    ┌──────────┐
 │  Active  │ ───────────────▶ │  Locked  │
 └──────────┘                  └──────────┘
      ▲
      └── 로그인 성공 → count = 0 (리셋)
```

"5회"라는 임계값은 결국 전이 조건, 즉 `count >= 5`라는 술어(predicate)다. 상태와 전이를 머릿속이 아니라 명시적인 변수(`loginFailedAttempts`, `accountStatus`)로 두니까, 인증 흐름을 그대로 상태 다이어그램으로 그릴 수 있었다. 면접에서 인증 로직을 설명할 일이 있으면 이 그림 한 장으로 정리되겠다 싶었다.

마침 얼마 전에 쓴 [모듈러 편향 글]({% post_url 2026-06-16-modulo-bias-maplestory %})도 결국 같은 결이었다. "값을 어떻게 이산적으로 쪼개고 경계를 긋느냐"가 핵심이었으니, 이산수학의 사고가 실제 버그로 나타난 사례였던 셈이다.

## 마무리 — 컴퓨터는 결국 이산적인 기계다

한 학기 해보고 제일 크게 남은 건, 컴퓨터가 연속적인 값을 다루는 기계가 아니라 **이산적인(discrete) 기계**라는 감각이다. 값을 명확히 쪼개고, 경계를 긋고, 조건을 빠짐없이 따져 예외 없는 로직을 만드는 것. 이게 이산수학이 내내 말하던 거고, 탄탄한 백엔드를 짜는 데도 같은 사고가 필요하다는 걸 이제야 좀 알 것 같다.

전공 첫 단추는 끼웠다. 다음 학기 과목들도 그냥 외우고 끝내는 게 아니라, 이렇게 실무랑 엮어가며 내 걸로 만들어볼 생각이다.

> 이론이 거창해 보여도 결국은, 내가 감으로 하던 걸 **이름 붙여 정리해주는** 도구다. 그 이름표가 생기면 다음 설계가 빨라진다.
